### B-01: Indent Validation Chain

- **Component A**: CLI `cmd_splice` calls `needs_indent_fix()` → `auto_indent_content()` → `validate_indentation()` at `src/main.rs:349-372`. These are in `src/indent.rs` (bin-crate mod, not pub in lib). CLI gates the write on indent validation: if `!valid && !dry_run`, returns error at `main.rs:360-369`.
- **Component B**: Python `sniper_edit` at `sniper-py/src/lib.rs:52-126`. Takes `content: &str`, directly calls `write_atomic(filepath, &refs)` at line 105. Zero indent imports, zero indent calls. Same for `sniper_manifest` at lines 172-248 (no indent logic).
- **Interaction**: Python passes raw content string to `write_atomic` which writes to disk via `write_atomic_impl` (`lib.rs:431-486`). No indent validation occurs between content receipt and disk write. Content with arbitrary indentation (0-space in a 4-space file, mixed tabs/spaces) is written verbatim.
- **Verdict**: VIOLATION — C7 (Semantic Information Loss). The indent validation capability exists in the codebase but is not transmitted to the Python write path. Source (user) intends a well-indented edit; boundary (Python→write_atomic) drops the indent-check capability entirely.
- **Severity**: HIGH — silently writes malformed indentation. Downstream tools (linters, compilers) reject the file; user has no warning.

### B-02: DAL Enforcement Bypass

- **Component A**: CLI uses `write_atomic_with_dal(filepath, &lines_refs, &guard, config.dal_level)` at `src/main.rs:427`. When `SNIPER_DAL_LEVEL=Maximum`, this performs `guard.check()` TWICE (`lib.rs:388-395`). The CLI also creates a `ResourceGuard::auto(0.5)` at `main.rs:382` and passes it explicitly.
- **Component B**: Python `sniper_edit` calls `write_atomic(filepath, &refs)` at `sniper-py/src/lib.rs:105`. `write_atomic` (`lib.rs:327-330`) calls `write_atomic_impl` directly — no guard parameter, no DAL check. A NEW guard is created inside `write_atomic_impl` (`lib.rs:467`) with `ResourceGuard::auto(0.5)`, but this is a POST-write metabolic check, NOT a pre-gate DAL enforcement.
- **Interaction**: User sets `SNIPER_DAL_LEVEL=Maximum` (env). CLI path: double `guard.check()` before rename. Python path: single `guard.check()` inside `write_atomic_impl` AFTER the BufWriter has flushed to the temp file. The DAL Maximum double-check never fires from Python. The promise (Maximum = extra validation) is silently voided.
- **Verdict**: VIOLATION — C5 (Capability Downgrade). The DAL capability is stripped at the Python→Rust boundary. User's security posture (Maximum) is downgraded to Baseline without notification.
- **Severity**: HIGH — security promise silently voided.

### B-03: Context Verification Omission

- **Component A**: CLI `cmd_splice` at `src/main.rs:311-315`: `if let Some(expected) = context_hash { verify_context(&lines, start, end, expected) }`. `verify_context` (`lib.rs:490-524`) hashes 3 lines before start and 3 lines after end, compares to expected SHA-256 prefix. Rejects edits when surrounding context has changed (stale line numbers).
- **Component B**: Python `sniper_edit` at `sniper-py/src/lib.rs:52-58`: signature is `fn sniper_edit(py, filepath, start, end, content)`. No `context_hash` parameter. No call to `verify_context`. The function `verify_context` IS pub in `lib.rs` — it could be imported but is NOT.
- **Interaction**: A Python user who computed a context_hash (e.g., from a prior read) has no way to pass it. The edit is applied against whatever the file currently contains, even if it changed since the line numbers were computed. Stale-context edits (the exact scenario `verify_context` prevents) are silently applied.
- **Verdict**: VIOLATION — C7 (Semantic Information Loss). The context verification intent is not carried across the Python→Rust boundary.
- **Severity**: MEDIUM — stale-context edits produce wrong results without error.

### B-04: Hex Encoding / Trust Model Divergence

- **Component A**: CLI requires hex-encoded content: `sniper <file> <start> <end> <hex>` (`main.rs:148-161`). Hex is decoded via `hex_decode(hex)` before use. The hex requirement is a SHELL-SAFETY boundary — hex bytes [0-9a-f] cannot be corrupted by shell metacharacters (`$`, `` ` ``, `|`, etc.). The CLI docstring explicitly states: "Hex encoding guarantees zero shell corruption" (`main.rs:3`).
- **Component B**: Python `sniper_edit` at `sniper-py/src/lib.rs:52-58` takes `content: &str` (plain text). No hex encoding needed because PyO3 FFI passes Python strings directly to Rust via C FFI — no shell intermediary. This is architecturally correct.
- **Interaction**: The hex encoding in CLI is a shell-protection mechanism, not a content-validation mechanism. Python bypasses the shell entirely, so hex encoding is unnecessary. The trust model is DIFFERENT (CLI: shell-untrusted→hex-encode→decode; Python: FFI-direct→no encoding). Both paths produce the same decoded content at the write boundary. No data is lost or corrupted.
- **Verdict**: CLEAN — C1 N/A (different trust models, same payload at write boundary). C7 N/A (no semantic info lost — hex is an encoding, not additional information).
- **Severity**: N/A — this is a non-issue. The hex invariant is shell-specific, not a content contract.

### B-05: Error Handling — Backtrack Signal Drop

- **Component A**: CLI at `src/main.rs:307`: `Err(e) => err(handle_backtrack_error(e, "Read"))`. At `lib.rs:484`: `Err(e) => Err(handle_backtrack_error(e, "Atomic write"))`. `handle_backtrack_error` (`lib.rs:567-573`) checks `e.raw_os_error() == Some(-7)` and produces a distinctive CRITICAL message about llmosafe Backtrack Signal.
- **Component B**: Python `sniper_edit` at `sniper-py/src/lib.rs:66`: `fs::read_to_string(filepath).map_err(|e| format!("read {filepath}: {e}"))`. At line 105: `write_atomic(filepath, &refs)?` — the `?` propagates the String error from `write_atomic`, which INTERNALLY calls `handle_backtrack_error` (`lib.rs:484`). BUT: the read path at line 66 does NOT call `handle_backtrack_error`.
- **Interaction**: For WRITE errors: `write_atomic` internally calls `handle_backtrack_error` at `lib.rs:482-484`, so the backtrack message IS present. However, the read path (`lib.rs:66`) formats the error directly without backtrack handling. If OS error -7 fires on a READ operation through Python, user gets raw IO error, not the backtrack message.
- **Verdict**: PARTIAL — C7 (partial semantic loss on read path only). Write path is CLEAN (internal call handles it). Read path drops the backtrack context.
- **Severity**: LOW — OS error -7 is a defensive fallback; primary resource exhaustion surfaces via `ResourceGuard::check()`, not OS signals.

### B-06: Security Policy — base_dir Always None

- **Component A**: `SecurityPolicy` at `src/security.rs:13-19` has `base_dir: Option<PathBuf>`. `validate_path` (`security.rs:64-107`) enforces base directory containment ONLY when `base_dir` is `Some`. CLI and lib both use `normalize_path` (`lib.rs:169-173`) which constructs `SecurityPolicy::default()` — `base_dir: None` (no containment).
- **Component B**: Python `sniper_edit` calls `normalize_path(filepath)` at `sniper-py/src/lib.rs:62`. Same default policy: `base_dir: None`.
- **Interaction**: Both CLI and Python use identical security policy (default, no base_dir). The `EscapesBaseDirectory` check (`security.rs:98-103`) is UNREACHABLE from either interface. This is NOT a Python-specific issue — it's a systemic gap shared by both paths. The `normalize_path_secure` function (which accepts base_dir) is dead code.
- **Verdict**: CLEAN (for the Python↔Rust boundary specifically — both sides have the SAME behavior). The base_dir gap is a structural issue (CAM F-12), not a cross-boundary violation.
- **Severity**: N/A for cross-boundary. (Systemic: LOW per CAM.)

### B-07: Risk Telemetry / Recommended Action Drop

- **Component A**: CLI `cmd_splice` at `main.rs:382-384` creates `ResourceGuard::auto(0.5)`, computes `RiskTelemetry::from_guard(&guard)`, and calls `recommend_from_risk(&risk)` to produce `recommended_action`. These are serialized into JSON output (`main.rs:460-461`). `CliResult` struct has `risk: Option<RiskTelemetry>` and `recommended_action: Option<String>`.
- **Component B**: Python `sniper_edit` at `sniper-py/src/lib.rs:111-125` builds `PyDict` with keys: `status`, `lines_removed`, `lines_inserted`, `total_lines`, `backup_path`. NO `risk`, NO `recommended_action`, NO `line_shift`, NO `ai_hint`. The `RiskTelemetry` struct is pub in `lib.rs:47-55` but never imported by the Python binding.
- **Interaction**: Rust generates risk telemetry (entropy, classifier_score, memory pressure). The CLI consumer sees `"risk": {...}` and `"recommended_action": "..."` in JSON. The Python consumer receives a dict with NONE of these fields. The user has no visibility into resource pressure or system-recommended action.
- **Verdict**: VIOLATION — C7 (Semantic Information Loss). Risk assessment is computed, then silently dropped at the boundary. Source has information (system is under pressure, consider pausing) that the sink (Python user) needs for informed decisions.
- **Severity**: MEDIUM — user cannot make resource-aware decisions from Python.

### B-08: Backup Counting / Manifest Promotion

- **Component A**: CLI at `main.rs:434-436`: `count_recent_backups(filepath, config.lock_timeout.as_secs())` → if count >= 3, `ai_hint` suggests `"Multiple edits to this file. Consider batching with manifest."`. `count_recent_backups` is pub at `lib.rs:528-554`.
- **Component B**: Python `sniper_edit` never calls `count_recent_backups`. The result dict has no `ai_hint` field. A Python user making 10 sequential edits never receives the manifest promotion hint.
- **Interaction**: The promotion signal (≥3 recent backups → suggest batching) is lost. Python users performing repeated single edits are never guided toward more efficient batch operations.
- **Verdict**: VIOLATION — C7 (Semantic Information Loss). The manifest promotion hint exists but doesn't cross the Python boundary.
- **Severity**: LOW — UX degradation, not security or correctness issue.

### B-09: DalLevel::from_env — Case-Sensitivity Promise Violation

- **Component A**: `DalLevel::from_env()` at `src/config.rs:30-36`. Docstring (`config.rs:28-29`): "Accepts 'Baseline', 'Enhanced', 'Maximum' (case-insensitive)." Implementation: `match env::var("SNIPER_DAL_LEVEL").as_deref() { Ok("Enhanced") => Self::Enhanced, Ok("Maximum") => Self::Maximum, _ => Self::Baseline }`.
- **Component B**: This is an INTERNAL Rust boundary (env var → Rust match). Both CLI and Python are affected equally since both call `SniperConfig::from_env()` which calls `DalLevel::from_env()`.
- **Interaction**: User sets `SNIPER_DAL_LEVEL=maximum` (lowercase, reasonable). Match arm `_ => Self::Baseline` fires. DAL level silently becomes Baseline. Only exact `"Enhanced"` and `"Maximum"` (case-exact) are recognized. The docstring's "case-insensitive" claim is violated.
- **Verdict**: VIOLATION — C1 (Contract Mismatch). The documented contract ("case-insensitive") does not match the implementation (case-exact). This is internal to Rust but compounds with B-02: Python user sets `SNIPER_DAL_LEVEL=maximum`, gets Baseline, AND Python skips DAL enforcement anyway.
- **Severity**: MEDIUM — non-default DAL levels silently ignored. Compounds with B-02.

### B-10: Module Visibility — indent.rs / diff.rs Are Bin-Crate Only

- **Component A**: `src/indent.rs` and `src/diff.rs` are declared `mod indent; mod diff;` in `src/main.rs:37-38`. They are NOT `pub mod` and NOT in `lib.rs`. They are accessible ONLY from the binary crate.
- **Component B**: Python binding (`sniper-py/src/lib.rs`) depends on `moesniper` (the lib crate), imported at line 7: `use moesniper::{...}`. It CANNOT import `indent` or `diff` modules because they are not part of the lib crate's public API.
- **Interaction**: This is the ROOT CAUSE of B-01 and partial B-07. The indent validation and diff preview logic is structurally inaccessible from the Python binding. Even if the Python binding WANTED to call `needs_indent_fix` or `validate_indentation`, it cannot — the functions don't exist in the `moesniper` namespace.
- **Verdict**: VIOLATION — C7 (Structural Semantic Information Loss). The module visibility boundary prevents the indent/diff capabilities from reaching Python. This is NOT a missing call — it's a missing EXPORT.
- **Severity**: HIGH — the root cause of the most impactful gaps (indent validation, dry-run preview).

## PHASE 2: CONCRETE WITNESS CONSTRUCTION

### Witness: B-02 DAL Enforcement Bypass (most dangerous)

**Scenario**: User sets `SNIPER_DAL_LEVEL=Maximum` expecting double resource validation.

**Trace from Python user → file written without DAL:**

1. User: `os.environ["SNIPER_DAL_LEVEL"] = "Maximum"`
2. User calls: `sniper.edit("target.py", 5, 5, "new_content")`
3. B2 facade (`__init__.py:46`): delegates to `sniper_edit`
4. B1 PyO3 (`sniper-py/src/lib.rs:59`): `let config = SniperConfig::from_env()` — reads `SNIPER_DAL_LEVEL`, calls `DalLevel::from_env()`
5. **B-09 triggers**: If value is `"maximum"` (lowercase), `DalLevel::Baseline` returned. If `"Maximum"` (exact), `DalLevel::Maximum` returned. Assume exact match succeeds: `config.dal_level == DalLevel::Maximum`.
6. B1 reads file, validates range, splices lines (`lib.rs:62-102`)
7. B1 creates backup (`lib.rs:95`)
8. B1 calls `write_atomic(filepath, &refs)` at `lib.rs:105`
9. **CRITICAL**: `write_atomic` (`src/lib.rs:327-330`) does NOT accept `dal_level` parameter. Calls `write_atomic_impl` directly.
10. `write_atomic_impl` creates guard internally (`lib.rs:467`), calls `guard.check()` once, then renames (`lib.rs:482`).
11. DAL Maximum's SECOND `guard.check()` never fires.

**Result**: File written. DAL enforcement bypassed. User's security posture silently downgraded. Config read the correct value but never used it.

**Concrete illegal state**: `config.dal_level == DalLevel::Maximum` AND only one `guard.check()` executed.

### Witness: B-01 Indent Validation Bypass

**Scenario**: Python user edits a 4-space-indented Python file with unindented content.

**Trace:**

1. Target file `app.py`: `def foo():\n    pass\n` (4-space indent)
2. User calls: `sniper.edit("app.py", 2, 2, "print('hello')")` — no indentation
3. B1 PyO3 (`sniper-py/src/lib.rs:59-105`): reads file, validates range, does NOT call any indent function
4. Content `"print('hello')"` (0 leading spaces) is spliced directly
5. `write_atomic` writes: `def foo():\nprint('hello')\n`
6. Result: `IndentationError` when Python executes `app.py`

**CLI contrast** (same edit):
1. `sniper app.py 2 2 $(sniper encode "print('hello')")`
2. `cmd_splice` calls `validate_indentation` → returns `(false, Some("INDENTATION WARNING: Replacement has 0 leading spaces, expected 4"))`
3. Since `!valid && !dry_run` → returns error, file NOT modified
4. User sees: `"Indentation validation failed: INDENTATION WARNING: ..."`

**Result**: CLI protects user from indent mistake. Python writes broken file silently.

## PHASE 3: CONVERGENT COVERAGE (Simulated backward trace)

### B-02 Backward Trace: From missing DAL check → Python binding

Starting from `write_atomic_impl`'s single `guard.check()`:
1. Who calls `write_atomic_impl`? → `write_atomic` and `write_atomic_with_dal` and `write_atomic_with_guard`
2. `write_atomic_with_dal` adds the second check (`lib.rs:391-395`). Who calls it? → `main.rs:427` (CLI only)
3. `write_atomic` (no DAL). Who calls it? → `sniper-py/src/lib.rs:105` and `:227` (Python only)
4. Backward trace confirms: Python ALWAYS routes through `write_atomic` (no DAL), CLI ALWAYS routes through `write_atomic_with_dal` (with DAL).

**Convergent confirmation**: Forward trace (Python→write_atomic→no DAL) and backward trace (write_atomic_impl←write_atomic←Python) both confirm the same finding.

### B-01 Backward Trace: From missing indent check → module visibility

Starting from a Python edit with bad indentation reaching disk:
1. Why wasn't indent checked? → No indent function called in `sniper_edit`
2. Why no indent function? → `needs_indent_fix`, `validate_indentation`, `auto_indent_content` are in `indent.rs`
3. Is `indent.rs` accessible? → `mod indent;` in `main.rs:37` — bin-crate only, NOT `pub mod` in `lib.rs`
4. Could Python binding add indent? → NOT without either (a) promoting `indent.rs` to `pub mod` in lib, or (b) duplicating indent logic in PyO3 binding
5. `lib.rs` does NOT re-export any indent functions.

**Convergent confirmation**: The root cause is structural (module visibility), not incidental (forgot to call).

## PHASE 4: COMPOUND CASCADE DETECTION

### CASCADE 1: B-09 + B-02 → Complete DAL Nullification (CHAINED)

```
B-09: SNIPER_DAL_LEVEL=maximum (lowercase) → DalLevel::Baseline (case-exact mismatch)
  │  User intends Maximum, gets Baseline silently
  ▼
B-02: Even if DalLevel::Maximum correctly parsed → Python calls write_atomic (no DAL param)
  │  DAL enforcement bypassed regardless of config value
  ▼
Result: DAL is COMPLETELY non-functional from Python. 
  - Case 1: Wrong case → Baseline from config, no enforcement anyway
  - Case 2: Right case → Maximum in config, still no enforcement
  BOTH cases produce identical behavior: single guard.check(), no double-check.
```

**Classification**: CHAINED + DUAL-DECISION. B-09 (parsing bug) and B-02 (missing wiring) independently nullify DAL from different angles. Fixing one leaves the other active.

### CASCADE 2: B-10 + B-01 → Structural Indent Bypass (CHAINED)

```
B-10: indent.rs is bin-crate mod → NOT accessible from Python binding
  │  Python binding CANNOT call indent functions even if it wanted to
  ▼
B-01: sniper_edit has no indent validation → writes any indentation verbatim
  │  File written with broken indent, no warning, no error
  ▼
Result: Python users structurally CANNOT access indent safety.
  The feature exists, is tested, is documented — but lives behind a visibility wall.
```

**Classification**: CHAINED. B-10 (structural) causes B-01 (behavioral).

### CASCADE 3: B-02 + B-07 → Invisible Risk (PARALLEL)

```
B-02: DAL enforcement bypassed → no double resource check
  │  
B-07: RiskTelemetry dropped → user cannot see resource pressure
  │
Result: System under resource pressure AND user has no visibility.
  CLI user sees risk JSON and recommended_action. Python user sees neither.
  Combined: Python operates blind to resource conditions that CLI monitors.
```

**Classification**: PARALLEL. Independent violations at the same boundary that compound into total situational blindness.

### CASCADE 4: B-09 + B-02 + B-03 → Silent Stale-Context Edit Under Maximum DAL (CHAINED)

```
B-09: User sets SNIPER_DAL_LEVEL=Maximum (exact case) → config.dal_level = Maximum
B-02: Python calls write_atomic (no DAL) → Maximum never enforced
B-03: No context_hash parameter → verify_context never called
  │
  │ Combined scenario:
  │ 1. User set Maximum DAL expecting highest safety
  │ 2. File changed between read and edit (another process / concurrent editor)
  │ 3. CLI would reject: verify_context detects context mismatch
  │ 4. Python: no context check AND no DAL double-check → stale edit applied
  ▼
Result: A stale-context edit is silently written to disk, with the user
  believing Maximum security is active. THREE safety layers bypassed.
```

**Classification**: CHAINED. Each violation enables the next. This is the WORST cascade.
