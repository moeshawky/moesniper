## ROOT CAUSE 1: Module Visibility ‚Äî indent.rs/diff.rs Are Bin-Crate Only

**Root Cause ID**: RC-01
**Line number**: `src/main.rs:36-37`
**Module**: `main.rs` (binary crate)

**Assumption**: "indent.rs and diff.rs are CLI-internal implementation details ‚Äî Python binding doesn't need them."

**Violation evidence**:
- `src/main.rs:36-37`: `mod diff; mod indent;` ‚Äî declared as bin-crate-private modules
- `src/lib.rs`: NO `pub mod indent;` or `pub mod diff;` ‚Äî not re-exported from lib crate
- `src/main.rs:349-372`: CLI uses `needs_indent_fix()`, `auto_indent_content()`, `validate_indentation()` from indent.rs
- `sniper-py/src/lib.rs:52-126`: Python `sniper_edit` has ZERO indent imports, ZERO indent calls
- CBP B-10: Python binding structurally CANNOT reach indent.rs ‚Äî it's not in the lib crate's public API

**Call chain** (4 hops):
```
main.rs:37 (mod indent) 
  ‚Üí indent.rs functions exist but are bin-crate-private
  ‚Üí lib.rs does not re-export indent functions
  ‚Üí Python binding (depends on lib crate) cannot import indent
  ‚Üí sniper_edit writes content with arbitrary indentation (B-01 violation)
  ‚Üí File written with broken indent, no warning (CAM F-01)
```

**CAM findings explained**: F-01, F-11 (Python missing indent params)
**CBP violations explained**: B-10 (structural root), B-01 (behavioral symptom)
**Fix direction**: Promote `indent.rs` to `pub mod` in `lib.rs`, re-export indent functions, add params to Python `sniper_edit`

## ROOT CAUSE 2: Temporal Drift ‚Äî write_atomic_with_dal Added After Python Binding Written

**Root Cause ID**: RC-02
**Line number**: `src/lib.rs:381-395` (write_atomic_with_dal), `sniper-py/src/lib.rs:105` (calls write_atomic)
**Module**: `lib.rs` + `sniper-py/src/lib.rs`

**Assumption**: "Python binding was written when `write_atomic` was the only write function ‚Äî `_with_dal` was added later, Python never updated."

**Violation evidence**:
- Git log: `482f618 Add risk telemetry, DAL gating, and PID pacing` ‚Äî DAL added as a later feature
- `src/lib.rs:381-395`: `write_atomic_with_dal` performs double `guard.check()` when `DalLevel::Maximum`
- `src/main.rs:427`: CLI uses `write_atomic_with_dal` (updated when DAL added)
- `sniper-py/src/lib.rs:105`: Python still calls `write_atomic` (never updated)
- CBP B-02: Python calls `write_atomic` ‚Üí no DAL enforcement, even when `SNIPER_DAL_LEVEL=Maximum`

**Call chain** (4 hops):
```
lib.rs:381 (write_atomic_with_dal added)
  ‚Üí CLI updated to use _with_dal (main.rs:427)
  ‚Üí Python binding NOT updated (still calls write_atomic at lib.rs:105)
  ‚Üí DAL Maximum config read but ignored (CBP B-09 compounds this)
  ‚Üí Python writes file without double resource check (CAM F-02)
```

**CAM findings explained**: F-02 (Python bypasses DAL)
**CBP violations explained**: B-02 (DAL bypass), CASCADE 1 + CASCADE 4 (dual-decision with B-09)
**Fix direction**: Switch Python `sniper_edit` to call `write_atomic_with_dal`, pass `config.dal_level`

## ROOT CAUSE 3: Documentation Lie ‚Äî DalLevel::from_env Claims Case-Insensitive, Implementation Case-Exact

**Root Cause ID**: RC-03
**Line number**: `src/config.rs:28-29` (docstring) vs `src/config.rs:30-36` (implementation)
**Module**: `config.rs`

**Assumption**: "The docstring says 'case-insensitive' ‚Äî someone assumed the implementation matches."

**Violation evidence**:
- `src/config.rs:28-29`: Docstring: "Accepts 'Baseline', 'Enhanced', 'Maximum' (case-insensitive)."
- `src/config.rs:30-36`: Implementation: `match env::var(...).as_deref() { Ok("Enhanced") | Ok("Maximum") => ..., _ => Baseline }`
- NO `.to_lowercase()` or `.to_uppercase()` before match ‚Äî case-exact matching
- Line 150 in config.rs has `.to_uppercase()` for something else ‚Äî proves author knows the pattern, didn't apply here
- CBP B-09: `SNIPER_DAL_LEVEL=maximum` (lowercase) silently becomes `Baseline`

**Call chain** (3 hops):
```
config.rs:30-36 (case-exact match)
  ‚Üí User sets SNIPER_DAL_LEVEL=maximum (lowercase, reasonable)
  ‚Üí Match arm _ => Baseline fires silently
  ‚Üí DAL Maximum never enforced (compounds with B-02)
  ‚Üí User believes Maximum security active, gets Baseline (CAM F-09)
```

**CAM findings explained**: F-09 (case-sensitivity orphan)
**CBP violations explained**: B-09 (case mismatch), CASCADE 1 + CASCADE 4 (dual-decision with B-02)
**Fix direction**: Add `.to_uppercase()` before match, or change docstring to "case-exact"

### Primary Structural Root

**RC-01 (module visibility)** is the PRIMARY structural root ‚Äî it's the only one that makes features STRUCTURALLY INACCESSIBLE. Even if you wanted to add indent validation to Python, you CAN'T without first promoting `indent.rs` to `pub mod` in `lib.rs`.

**RC-02 + RC-03** form a DUAL-DECISION cascade ‚Äî fixing either alone is insufficient. Both must be fixed to restore DAL enforcement from Python.

### Resolution Strategy

**Minimal fix set** (resolves all cascades):
1. Promote `indent.rs` and `diff.rs` to `pub mod` in `lib.rs`, re-export functions
2. Switch Python `sniper_edit` to call `write_atomic_with_dal`, pass `config.dal_level`
3. Fix `DalLevel::from_env` to use `.to_uppercase()` before match
4. Add `auto_indent`, `force_indent`, `dry_run`, `context_hash` params to Python `sniper_edit`
5. Add `risk`, `recommended_action`, `ai_hint` fields to Python result dict

**One sentence**: Promote bin-crate modules to lib, update Python binding to call the DAL-enforcing write function with all safety params, and fix the case-sensitivity lie.
