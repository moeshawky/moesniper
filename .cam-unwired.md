## FINDINGS

### F-01 — Python `sniper_edit` exposes ZERO indent/context/dry-run params
- **File:Line**: `sniper-py/src/lib.rs:52-58`
- **Feature/Flag**: `auto_indent`, `force_indent`, `dry_run`, `context_hash`
- **Evidence**: Rust `cmd_splice` signature (`src/main.rs:278-287`) takes 8 params (filepath, start, end, content, dry_run, auto_indent, force_indent, context_hash). Python `sniper_edit` takes only 4 (filepath, start, end, content). The indent module (`src/indent.rs`) is wired into CLI via `mod indent;` + `use indent::{auto_indent_content, needs_indent_fix, validate_indentation}` (main.rs:37-40) but the Python `sniper_edit` body never imports or calls any indent/diff/context function.
- **Wiring Status**: UNWIRED (Python)
- **Severity**: HIGH — Python users cannot access indentation validation, auto-fix, dry-run preview, or context-hash verification.

### F-02 — Python `sniper_edit` bypasses DAL enforcement
- **File:Line**: `sniper-py/src/lib.rs:105`
- **Feature/Flag**: `SNIPER_DAL_LEVEL` (env var, config.rs:114-115)
- **Evidence**: CLI uses `write_atomic_with_dal(filepath, &lines_refs, &guard, config.dal_level)` (main.rs:427). Python binding calls `write_atomic(filepath, &refs)` (sniper-py/src/lib.rs:105) — no guard, no DAL. Even if user sets `SNIPER_DAL_LEVEL=Maximum`, Python path ignores it.
- **Wiring Status**: PARTIALLY WIRED (reads config, ignores `dal_level` field)
- **Severity**: HIGH

## CASCADE DETECTED

**S-UNWIRED + G-SURFACE + S-ORPHAN** (at Python binding boundary)

Diagnosis: The Rust library implements a rich surface (indent validation, DAL enforcement, context verification, risk telemetry, dry-run previews). The Python binding wraps a **strict subset** — it calls `write_atomic` instead of `write_atomic_with_dal`, skips every indent/diff/context function, and strips risk telemetry from output. CLI users see full capabilities; Python users see a degraded facade.

The S-ORPHAN on `DalLevel::from_env` (F-09) compounds: Python users who DO set `SNIPER_DAL_LEVEL=Maximum` expect double-check enforcement — Python silently skips it.

Bridge rule triggered: **structural audit complete — Python binding is a missing-wiring failure, not a correctness bug. AD/AP/CBP cannot trace the absent call chain. Fix is wiring, not debugging.**

## MODULE REACHABILITY MATRIX

| Module | Declared pub | CLI reach | Python reach | lib.rs re-export |
|---|---|---|---|---|
| config.rs | Y (pub mod) | Y (from_env) | Y (sniper_config) | Y (SniperConfig, DalLevel) |
| security.rs | Y (pub mod) | Y (normalize_path wraps validate_path) | Y (normalize_path wraps) | Y (all 4 types) |

| indent.rs | N (mod only, main) | Y (--auto/--force-indent) | **N** | N (not re-exported) |

`diff.rs` and `indent.rs` are `mod` (crate-local to main binary), NOT `pub mod` in lib.rs. They are unreachable from Python *by design of module visibility* — Python can only call through `moesniper::` (the lib crate). This is the ROOT CAUSE of F-01, F-11: the indent/diff logic lives in the BIN crate, not the LIB crate, so the Python binding (which depends on the lib crate) structurally cannot reach it.
